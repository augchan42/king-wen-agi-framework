# Consolidated Code Review

Agents run: Diff, Holistic

## Critical

### 1. Invalid King Wen sequence corrupts the analysis
- File: `paper/generate_plots.py:92-103`
- Severity: `critical`
- Category: `Diff`
- Description: `king_wen_sequence` is not a valid 64-item permutation. It contains only 54 unique hexagrams, with 10 duplicates and 10 missing states. That means the computed transition metrics, plots, and conclusions are based on the wrong dataset.
- Suggested fix: Replace the hard-coded sequence with a validated canonical ordering, and add an assertion or test that enforces `len(sequence) == 64` and `len(set(sequence)) == 64`.

## High

### 2. Random-baseline significance testing uses the wrong unit of analysis
- File: `paper/generate_plots.py:282-315`
- Severity: `high`
- Category: `Diff`
- Description: The script pools all 1,000 random permutations into one 63,000-sample transition array and compares that against the single 63-transition King Wen ordering with KS/Levene tests. That inflates the random baseline sample size and makes the reported p-values statistically unsound.
- Suggested fix: Compare King Wen against permutation-level summary statistics and report empirical percentiles or Monte Carlo p-values instead of pooled-sample hypothesis tests.

## Medium

### 3. `warmdown_ratio=0.0` can trigger division by zero despite being the documented ablation setting
- File: `experiment/king_wen_schedules.py:55-57`, `experiment/king_wen_schedules.py:86-87`, `experiment/king_wen_schedules.py:109-111`, `experiment/program.md:68-69`
- Severity: `medium`
- Category: `Diff`
- Description: The experiment instructions explicitly say to set `warmdown_ratio=0.0` for an ablation, but the learning-rate helpers divide by `warmdown_ratio` in the cooldown branch. That makes the documented configuration capable of raising `ZeroDivisionError`.
- Suggested fix: Treat `warmdown_ratio <= 0` as "no cooldown" and bypass the cooldown branch entirely in all three helpers.

### 4. Recent analysis and scheduling changes shipped without regression coverage
- File: `paper/generate_plots.py:1`, `experiment/king_wen_schedules.py:1`
- Severity: `medium`
- Category: `Diff`
- Description: There are no automated checks covering sequence integrity, output bounds, or schedule boundary behavior. That test gap is sufficient to let both the invalid King Wen sequence and the zero-warmdown failure mode land undetected.
- Suggested fix: Add focused tests for sequence completeness and uniqueness, metric output shapes and ranges, and LR scheduler behavior at boundary values such as `warmup_ratio=0`, `warmdown_ratio=0`, and `progress` at segment edges.

### 5. The repository lacks durable agent operating guidance
- Directory: `/`
- Severity: `medium`
- Category: `Holistic`
- Description: There are no checked-in `AGENTS.md` files anywhere in the repository and no `CLAUDE.md` compatibility symlinks. New agents only inherit ephemeral local state under `.claude/`, so they do not get stable instructions about directory purpose, safe edit boundaries, validation steps, or common pitfalls.
- Suggested fix: Add a root `AGENTS.md` and focused `AGENTS.md` files in major working directories such as `paper/` and `experiment/`. Add `CLAUDE.md` symlinks beside them if Claude Code compatibility is required.

### 6. The README misstates the project layout and entrypoints
- File: `README.md`
- Severity: `medium`
- Category: `Holistic`
- Description: The README refers to a nonexistent `src/` directory and points build instructions at `paper` / `paper.tex`, while the tracked TeX entrypoint is `paper/king-wen.tex`. That makes the repository harder to navigate and increases the chance that both humans and agents run the wrong commands.
- Suggested fix: Update the README to describe the actual top-level layout (`paper/`, `experiment/`, `.claude/`, `reviews/`) and correct the paper build instructions to the real TeX entrypoint and command sequence.

### 7. Python dependencies are implicit rather than reproducible
- File: `/`, `paper/generate_plots.py:14-19`
- Severity: `medium`
- Category: `Holistic`
- Description: The analysis script imports `numpy`, `matplotlib`, `scipy`, and `statsmodels`, but the repository has no `pyproject.toml`, `requirements.txt`, or equivalent dependency manifest. New developers and agents cannot reproduce the environment deterministically.
- Suggested fix: Add a single Python dependency manifest and document the install and execution commands in the README.

### 8. Experiment execution is tightly coupled to an external repository with no local guardrails
- Directory: `experiment/`
- Severity: `medium`
- Category: `Holistic`
- Description: The experiment workflow depends on an external `autoresearch` repository and manual copying steps, but this repo does not provide a setup wrapper, validation command, or explicit interface boundary for that integration. That makes execution brittle and agent changes risky.
- Suggested fix: Add a setup or check script that validates the external repo location, required files, and expected paths, or formalize the integration behind a clearer adapter boundary.

### 9. The analysis script is monolithic and mixes multiple concerns
- File: `paper/generate_plots.py`
- Severity: `medium`
- Category: `Holistic`
- Description: The main analysis module combines static dataset definition, metric computation, statistical testing, and plotting in one file. That structure makes targeted changes, reuse, and isolated testing unnecessarily difficult.
- Suggested fix: Split the module by responsibility, for example into `data`, `metrics`, `stats`, and `plots`, and keep the top-level entrypoint focused on orchestration.

## Low

### 10. Core Python modules have minimal type coverage and no baseline test suite
- Directory: `paper/ and experiment/`
- Severity: `low`
- Category: `Holistic`
- Description: The Python code has very limited type annotations and no automated test suite. That is manageable for a small research repository, but it lowers confidence when changing schedule logic or analysis code.
- Suggested fix: Add type hints to public functions in `paper/generate_plots.py` and `experiment/king_wen_schedules.py`, then create a minimal test suite covering metric calculations and schedule edge cases.

### 11. Runtime failures surface as raw exceptions with little validation
- File: `paper/generate_plots.py and experiment/king_wen_schedules.py`
- Severity: `low`
- Category: `Holistic`
- Description: Missing dependencies, malformed sequences, and invalid input values currently appear likely to surface as raw Python exceptions without much validation or user-facing guidance.
- Suggested fix: Add lightweight input validation and CLI-friendly error messages for missing packages, invalid sequence lengths, and out-of-range progress values.

### 12. Reproducibility and observability are mostly manual
- File: `experiment/program.md and paper/generate_plots.py`
- Severity: `low`
- Category: `Holistic`
- Description: Experiment tracking relies on handwritten `results.tsv` updates and the plotting script only emits simple `print` output. There is no standard run metadata capture for parameters, seeds, commit hashes, or generated outputs.
- Suggested fix: Introduce a small structured logging or run-metadata convention so experiment inputs and outputs are recorded automatically.

### 13. `.claude/` contains session state but not a reusable project harness
- Directory: `.claude/`
- Severity: `low`
- Category: `Holistic`
- Description: The `.claude` directory appears to hold local review state and permissions rather than durable repository guidance. That is useful for one session, but it does not help future agents understand the project consistently.
- Suggested fix: Keep ephemeral agent state separate from durable project instructions and move lasting guidance into checked-in docs such as `AGENTS.md`.
